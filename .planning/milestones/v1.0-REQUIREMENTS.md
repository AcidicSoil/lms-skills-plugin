# Requirements Archive: v1.0 Host/WSL Execution and Project Workspaces

**Status:** Complete
**Shipped:** 2026-07-13
**Core Value:** Users can safely run skill-driven file and shell workflows in one predictable project workspace across Windows Host and WSL execution.

## Validated Requirements

### Configuration and Detection

- [x] **EXEC-01**: Host execution remains the backward-compatible default.
- [x] **EXEC-02**: Windows users can select WSL execution when available.
- [x] **EXEC-03**: Installed WSL distributions can be viewed and selected.
- [x] **EXEC-04**: Missing WSL or distributions produce actionable errors.
- [x] **EXEC-05**: Existing persisted settings load without manual migration.

### Workspace Lifecycle

- [x] **WORK-01**: Each provider working directory resolves to a deterministic workspace identifier.
- [x] **WORK-02**: Active workspaces are created idempotently before project tool use.
- [x] **WORK-03**: Users can inspect mode, distribution, workspace root, and active command cwd.
- [x] **WORK-04**: Host uses Host-native roots; WSL uses Linux-native roots.
- [x] **WORK-05**: Stale workspaces or removed distributions fail safely.

### Paths and Tool Routing

- [x] **PATH-01**: Relative project paths resolve from the workspace root.
- [x] **PATH-02**: Windows, Linux, and WSL UNC path kinds are explicitly classified.
- [x] **PATH-03**: Path handling preserves spaces, Unicode, separators, and drive normalization.
- [x] **PATH-04**: Canonical containment rejects workspace escape.
- [x] **TOOL-01**: Project file tools share one workspace context.
- [x] **TOOL-02**: Commands execute in the same selected environment and workspace.
- [x] **TOOL-03**: Skill-library access remains governed by configured skill roots in the selected environment.

### Reliability, Safety, Tests, and Documentation

- [x] **SAFE-01**: WSL invocation uses argument-safe process APIs.
- [x] **SAFE-02**: Host and WSL commands enforce timeout and output limits.
- [x] **SAFE-03**: Timeout handling attempts descendant termination and reports uncertainty.
- [x] **TEST-01**: Settings, capability detection, path handling, and containment are tested.
- [x] **TEST-02**: Host/WSL workspace integration is tested.
- [x] **TEST-03**: Host compatibility, build, diagnostics, and release verification remain passing.
- [x] **DOCS-01**: Setup, modes, workspace locations, limitations, and troubleshooting are documented.

## Adjustments During Delivery

- “Per-chat” workspace identity is implemented using LM Studio's available provider working-directory identity because the SDK does not expose a stable chat identifier.
- Skill roots were made environment-aware so WSL mode resolves and reads Linux-native skill paths instead of Windows paths.
- `change_directory` was added as a persistent command-cwd tool while project file paths remain workspace-root scoped.
- WSL shell semantics were tightened to Bash; Windows Host gained explicit cmd, PowerShell, Git Bash, and custom-shell support.
